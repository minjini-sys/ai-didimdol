# 데모 시나리오

세 시나리오 모두 같은 흐름을 따릅니다: 입력 → 의도 파악 → GitHub Skill 후보 검색 → 승인 → 파일 임시 확인 → 결과 생성.

## 1. 긴 자료 붙여넣고 요약만 시키기

입력:

```text
MCP_Server.py는 센서 데이터 자동 수집, 서버 연결, 파일 저장, RAG 검색 기능을 포함한다.
이 시스템은 반복 작업 자동화와 데이터 저장을 통해 운영 효율을 높인다.
이거 요약해줘
```

포인트: 자료 안에 "MCP", "자동화", "RAG" 같은 단어가 많아도 마지막 명령이 "요약해줘"이므로 Intent Split이 이를 `material`로 분리하고, Router는 "요약과 핵심 정리"로 분류합니다.

예상 결과:

- intent: 요약과 핵심 정리
- 검색어: summarization skill, document summary prompt, key point extraction agent
- 결과물: 자료 전체를 그대로 붙여넣지 않고 핵심 요약 + 한 줄 요약으로 정리된 결과

## 2. 공모전 안내문 조건 정리

입력:

```text
긴 공모전 안내문을 읽고 핵심 조건과 제출물을 정리하는 데 도움이 되는 AI Skill을 찾고 싶어.
```

예상 결과:

- intent: 문서 읽기와 핵심 조건 정리
- GitHub에서 document summarization / requirement extraction 관련 저장소 후보 검색
- 후보 카드에 "다운로드하지 않고, 로컬에도 저장하지 않습니다" 정책이 항상 함께 표시
- 사용자가 후보를 승인하면 README를 임시로 읽어 관련성과 위험 신호를 재확인한 뒤 결과 생성

## 3. 위험 신호가 있는 후보 차단

입력:

```text
반복 업무를 자동화할 Skill을 찾고 싶어.
```

검색 결과에 "브라우저를 조작하고 터미널 명령을 실행할 수 있다"는 설명의 저장소가 섞여 있으면, `assessBeforeDownload`가 이를 `blocked`로 분류합니다.

예상 결과:

- 해당 후보는 승인 버튼 대신 "권한이 큰 도구일 수 있어 이 단계에서는 다운로드하지 않습니다" 안내만 표시
- 사용자가 승인하려고 해도 서버가 해당 id를 결과 생성에 포함시키지 않음

## 핵심 답변

AI 디딤돌은 ChatGPT처럼 바로 답을 만들지 않고, 사용자의 실제 명령을 자료와 분리해 정확히 읽은 뒤 어떤 Skill이 왜 필요한지 설명하고, 승인받은 것만 임시로 확인해 사용합니다. 비전공자도 전공자처럼 "이 작업엔 이런 도구가 맞다"는 판단 과정을 그대로 따라갈 수 있습니다.
